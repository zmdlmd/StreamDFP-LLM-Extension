# MC1 抽取失败原因与修复进展（2026-03-20）

## 1. 结论摘要

`mc1` 之前抽取失败的主要原因，不是模型本身先天失效，而是早期复用的 `pilot20k` 输入样本存在明显问题：

- 旧版 `window_text_mc1_pilot20k.jsonl` 是一份早期 `sequential` 顺序采样产物，不是当前应使用的分层采样产物。
- 该样本几乎全部集中在 `2018-01-01` 到 `2018-01-02`，窗口历史上下文严重不足。
- 结果是 `Phase1` 生成的 `summary_text` 几乎没有有效事件证据，导致 `Phase2` 输出大面积塌缩为 `unknown`。
- 因此旧版 `mc1` 的 `Phase3` 在当前严格口径下会大量出现 `kept=0`、`degenerate_skip`。

本轮已经通过重新生成 `mc1 stratified_v2` 输入修复了这个问题。修复后，三种模型在 `Phase2` 上都已经不再是 `20000/20000 unknown` 的退化状态，并且在当前严格 `phase3` 口径下都已经能复现 `mc1 +2.2296` 的正增益结果。

最终结论见：

- [mc1_stratified_v2_final_summary_20260323.md](/root/autodl-tmp/StreamDFP/docs/reports/mc1_stratified_v2_final_summary_20260323.md)

## 2. 旧版失败现象

旧版输入文件：

- `llm/framework_v1_mc1/window_text_mc1_pilot20k.jsonl`
- `llm/framework_v1_mc1/reference_mc1_pilot20k.json`

旧版样本的典型问题：

- 样本分布几乎只覆盖最早两天
  - `2018-01-01`: `107`
  - `2018-01-02`: `19893`
- `reference_mc1_pilot20k.json` 元信息显示其采样方式为 `sample_mode = sequential`
- `summary_text` 中大量出现：
  - `ALLOWED_EVENT_FEATURES: none`
  - `active_features=0/0 known_features=0`
  - `RULE_PRED: unknown`

这导致旧版 `Phase2` cache 出现统一退化：

- `root_cause_pred = unknown`
- `confidence = 0`
- `risk_hint = 0`
- `llm_q_score = 0`

进一步影响 `Phase3`：

- `build_cache_variant.py` 过滤后大量 `kept=0`
- 当前严格口径下直接 `degenerate_skip`

## 3. 为什么旧版会失败

根因可以概括为两点：

### 3.1 输入样本是错误的 pilot20k 口径

旧版 `mc1` 并不是现在期望的“按日期和 disk 分层抽样”，而是顺序采样。  
对 `30d` 窗口来说，这意味着大量窗口过早落在时间轴前端，没有足够历史观测用于构造稳定异常特征。

### 3.2 事件证据在 Phase1 就已经塌掉

一旦 `summary_text` 中没有有效事件特征，后面的模型无论是：

- `Qwen3-4B-Instruct-2507`
- `Qwen3.5-4B`
- `Qwen3.5-Plus`

都会更倾向于输出 `unknown` 或极低置信度结果。  
因此旧版 `mc1` 的问题本质上首先是 `Phase1` 输入问题，而不是单纯模型优劣问题。

## 4. 修复方案

本轮重新构造了新的 `mc1` 输入：

- `llm/framework_v1_mc1/window_text_mc1_pilot20k_stratified_v2.jsonl`
- `llm/framework_v1_mc1/reference_mc1_pilot20k_stratified_v2.json`
- `llm/framework_v1_mc1/reference_mc1_pilot20k_stratified_v2_quality.json`

修复策略：

- 采样方式改为 `stratified_day_disk`
- 输出时间范围调整为 `2018-02-01 ~ 2018-03-13`
- 避开最早一批几乎没有历史上下文的窗口
- 同时重建 `reference_pool`，避免继续复用坏参考样本

## 6. 关于 legacy 结果的解释

历史上 `mc1` 有一条 `+2.2296` 的提升记录，这条结果并不是伪造，但需要按旧口径理解。详见：

- `docs/diagnostics/mc1_legacy_repro_20260314.md`

关键点：

- 历史旧流程允许在 `kept=0` 时继续跑 `loader/simulate`
- 当前严格口径则会在 `kept=0` 时直接 `degenerate_skip`
- 因此历史 `mc1 +2.2296` 不能直接解释为“旧版 Phase2 语义抽取完全正常”

## 8. 建议

- 后续 `mc1` 实验应全部切换到 `pilot20k_stratified_v2` 输入，不再复用旧版 `pilot20k` 顺序采样文件。
- 当前 `mc1` 的结论已经从“旧输入下失败、只能用 legacy 解释”更新为“修正输入后，当前严格口径也可稳定得到正增益”。
- 若后续要做模型选择：
  1. API 方案优先考虑 `Qwen3.5-Plus`
  2. 本地强基线优先考虑 `Qwen3-4B-Instruct-2507`
  3. `Qwen3.5-4B` 不建议作为首选正式方案
